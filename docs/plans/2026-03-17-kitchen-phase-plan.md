**Date:** 2026-03-17

## Overview

Two independent phases, each shippable on its own:

| Phase | Name | Goal | Risk |
|---|---|---|---|
| **Phase K1** | Fluent Emoji Icons | Replace broken HF-generated sprites with polished 3D icons | Low |

## Phase K1 — Fluent Emoji Icon System

### Problem
- `FLUX.1-schnell` generates inconsistent, poorly-cropped 64×64 images — wrong aspect ratios, items cut off, style varies wildly between items
- The HF generation pipeline adds ~5–15s latency per new pantry item
- Only 3 sprites in DB currently, all require manual normalization fixes

### Solution
Replace the HF sprite generation pipeline with **Microsoft Fluent Emoji 3D PNGs** as the primary icon source.

Fluent Emoji provides 3D-rendered food icons that look like polished cozy-game assets. MIT license. 200+ food items. Pre-built, consistent style. Zero latency.

**Strategy:**
1. Build a static mapping from BubblyChef food names → Fluent Emoji PNG filenames
2. Serve icons from `web/public/food-icons/` (committed static assets)
3. Keep the `/api/sprites/` backend as an **optional override** — if a custom approved sprite exists in DB, it wins; otherwise Fluent Emoji; otherwise category emoji
4. Remove the auto-generation background task from `pantry.py` and `scan.py` (stop polluting DB with bad sprites)

### Implementation

#### Step 1 — Download Fluent Emoji food PNGs
```bash
scripts/download_fluent_emoji.py
```
- Fetch the Fluent Emoji release index from GitHub API: `https://api.github.com/repos/microsoft/fluentui-emoji/git/trees/main?recursive=1`
- Filter entries whose path matches `assets/*/3D/*.png`
- Download all food-relevant PNGs to `web/public/food-icons/fluent/`
- Filename convention: `{emoji-slug}.png` (e.g., `tomato.png`, `milk-glass.png`)
- ~100–150 food/drink emoji to download

#### Step 2 — Build static name→icon map
**File:** `bubbly_chef/domain/icon_map.py` (NEW)

```python
# Maps canonical food names → Fluent Emoji PNG filename (no path, no extension)
# Generated by scripts/build_icon_map.py; committed as static data
FOOD_ICON_MAP: dict[str, str] = {
    "tomato": "tomato",
    "potato": "potato",
    "milk": "glass-of-milk",
    "bread": "bread",
    "chicken": "poultry-leg",
    "salmon": "fish",
    "egg": "egg",
    "butter": "butter",
    "cheese": "cheese-wedge",
    # ... ~150 entries
}
```

**File:** `scripts/build_icon_map.py` (NEW)
- Reads `pantry_catalog.json` (304 entries)
- For each entry, uses the `emoji` field to look up the matching Fluent Emoji slug
- Writes `FOOD_ICON_MAP` to `bubbly_chef/domain/icon_map.py`

#### Step 3 — New `/api/icons/{item_name}` endpoint

**Rationale:** Keep icon serving backend-mediated so the frontend doesn't need to know the mapping details. The endpoint returns a redirect or the PNG directly.

```
GET /api/icons/{item_name}
→ 1. Check DB for approved custom sprite → serve BLOB
→ 2. Check FOOD_ICON_MAP for Fluent Emoji filename → 302 redirect to /food-icons/fluent/{slug}.png
→ 3. Fall back to category emoji slug → redirect to /food-icons/fluent/category/{category}.png
→ 4. 404 if nothing found
```

This means the frontend HEAD check at `/api/sprites/{name}` can be replaced with a simpler `src="/api/icons/{name}"` — always returns *something*.

#### Step 4 — Update frontend icon components

**`KitchenItem.tsx`**: Replace `useSpriteUrl` hook + HEAD check with a simple `<img src={/api/icons/${name}} />`. No more async HEAD ping, no more 404 fallback logic in the component.

**`Pantry.tsx` / `PantryItemIcon`**: Same simplification.

**Result:** Every item always shows an icon. No loading flicker, no HEAD-then-render delay.

#### Step 5 — Remove broken HF generation

- `pantry.py` `create_pantry_item`: remove `_generate_and_save` background task
- `scan.py` `confirm_items`: remove `_generate_and_save` background task
- Clear bad sprites from DB (one-time migration script)
- Keep `sprite_generator.py` and `/api/sprites/` routes but mark them as the **admin custom override path** only

## Phase K2 — Phaser.js Kitchen Game Scene

### Problem
The current DOM kitchen scene (`DOMKitchenScene.tsx`) is functional but limited:
- Items are placed in a fixed mathematical grid — no user agency
- No animations between states (items don't "move" when placed)
- Cannot support drag-and-drop item organization
- No progression/decoration system
- The 2D pixel-art vision has no path forward in DOM + CSS

### Solution
Replace `DOMKitchenScene.tsx` with a **Phaser 3 game scene** embedded in a React component.

Phaser 3 gives us:
- Tile-map based kitchen room (can define shelf zones as tile regions)
- Sprite system with proper texture atlases
- Drag-and-drop built-in (`Phaser.Input.Pointer` + drag events)
- Scene manager (Room → Interior as separate Scenes, not React state)
- Tweens for animations (item bouncing onto shelf, fridge opening)
- Game loop = items can animate (wiggle on hover, glow when expiring)

### Architecture

```
React Component: KitchenGame.tsx
  └── Phaser.Game (instance, created in useEffect)
       ├── Scene: KitchenRoom (main view)
       │    ├── TiledJSON map: kitchen-room.json
       │    ├── TextureAtlas: appliances.json + appliances.png
       │    ├── GameObjects: ApplianceZone (interactive)
       │    └── On click → launch KitchenInterior scene
       └── Scene: KitchenInterior (zoomed view)
            ├── TiledJSON map: {location}-interior.json
            ├── TextureAtlas: food-icons.json + food-icons.png
            ├── GameObjects: FoodItem (draggable)
            └── Drag → persist new position to DB via API
```

### Gamification Layer

#### Item Placement (User-controlled)
- Each interior has a **grid of placement slots** (defined in Tiled map as object layer)
- Items can be dragged between slots
- Position is persisted: `PATCH /api/pantry/{id}` with `slot_position` (new field)
- On load, items appear in their last-saved slot; new items animate in to first empty slot

#### Progression System
- **Decorations table** in SQLite: `user_decorations(id, decoration_id, unlocked_at, placed_at, slot)`
- Milestones trigger decoration unlocks (non-blocking, award on next visit):
  - 5 items in pantry → unlock "cute plant"
  - First receipt scan → unlock "recipe book on counter"
  - 30 items total → unlock "spice rack"
  - 7-day streak → unlock "hanging herb garden"
- Decorations are draggable in the kitchen, same as food items
- Visual: decoration sprites live in `web/public/kitchen/decorations/`

#### Expiry Signals (Game Feedback)
- Items expiring soon: gentle pulse animation (Phaser tween, scale 1.0→1.05→1.0 loop)
- Expired items: red tint overlay on sprite
- Freshness aura: subtle green glow on very fresh items (>7 days)

### Implementation Steps

- `npm install phaser`
- `KitchenGame.tsx`: mounts Phaser.Game in `useEffect`, destroys on unmount
- Pass pantry items as initial data via game config `scene.data`
- React → Phaser bridge: custom `EventEmitter` for pantry updates

#### K2.2 — Asset download script (automated, no design tool required)
**File:** `scripts/download_kitchen_assets.py` (NEW)

Downloads all art from free CC0 sources at build time:


Script is idempotent — skip download if file already exists. Uses `httpx` (already a dep).

#### K2.3 — Maps generated in code, not Tiled
**No Tiled editor required.** Maps are defined as TypeScript constants and rendered directly by Phaser using `Phaser.Tilemaps.Tilemap.createBlankDynamicLayer()`.

```typescript
// web/src/game/maps/kitchenRoomMap.ts
export const KITCHEN_ROOM_LAYOUT = {
  width: 25, height: 16, tileSize: 32,
  layers: {
    floor:    { tiles: FLOOR_TILE_DATA },    // checkerboard cream/beige
    walls:    { tiles: WALL_TILE_DATA },     // upper wall + wainscoting
    appliances: { zones: [
      { id: 'fridge',  x: 10, y: 4, w: 4, h: 6 },
      { id: 'freezer', x: 15, y: 4, w: 4, h: 6 },
      { id: 'pantry',  x: 2,  y: 3, w: 6, h: 8 },
      { id: 'counter', x: 4,  y: 9, w: 14, h: 4 },
    ]},
  },
};
```

Interior maps are similarly defined as `FRIDGE_INTERIOR_LAYOUT`, etc. — slot grid positions are arrays of `{x, y}` tile coordinates, computed programmatically from shelf count and item capacity.

This means iterating on layout = editing a TypeScript constant and hot-reloading. Zero round-trip through an external design tool.

#### K2.4 — Texture atlas for food icons (automated)
**File:** `scripts/build_phaser_atlas.py` (NEW)

Takes the Fluent Emoji PNGs from K1 (`web/public/food-icons/fluent/`) and packs them into a Phaser-compatible texture atlas using Python's `Pillow`:
- Strips alpha, resizes each to 32×32, arranges in rows of 16
- Outputs `web/public/kitchen/atlas/food-icons.png` + `food-icons.json` (Phaser atlas format)
- Frame keys = canonical food names matching `icon_map.py`

Script is fully automated — run it once after K1 is complete, re-run any time new icons are added.

#### K2.4 — KitchenRoom scene
- Load tilemap, render room
- Place `ApplianceZone` interactive areas over each appliance
- Item count badges via Phaser Text objects
- Click zone → launch matching Interior scene with slide transition

#### K2.5 — KitchenInterior scenes (4 × location)
- Load interior tilemap
- For each pantry item in this location: create `FoodItem` sprite at saved slot (or first empty)
- Enable drag: `setInteractive()` + `scene.input.setDraggable(item)`
- On drag end: snap to nearest valid slot, fire `PATCH /api/pantry/{id}` to save position
- Back button: stop Interior scene, resume Room scene

#### K2.6 — Progression system
- `GET /api/decorations` → returns unlocked decorations
- `POST /api/decorations/check` → checks milestones, awards new unlocks, returns new items
- Call on app load and after each pantry mutation
- Unlocked decorations appear in a "decorations drawer" panel (slide-up from bottom)
- Drag from drawer onto kitchen scene to place

#### K2.7 — Migration from DOM scene
- `KitchenView.tsx` swaps `DOMKitchenScene` for `KitchenGame` behind a feature flag (`VITE_KITCHEN_V2=true`)
- Keep DOM scene working in parallel during development
- Delete DOM scene + PixiJS after K2 ships and is validated

### New Backend Endpoints

```
GET  /api/decorations                    # list user's unlocked decorations + placed state
POST /api/decorations/check-milestones   # check for new unlocks (idempotent)
PATCH /api/pantry/{id}/position          # save item slot position {location, slot_index}
```

### New DB Tables

```sql
CREATE TABLE user_decorations (
    id TEXT PRIMARY KEY,
    decoration_id TEXT NOT NULL,
    unlocked_at TEXT NOT NULL,
    placed_location TEXT,     -- null = in drawer, else fridge/pantry/counter/freezer
    slot_index INTEGER        -- position within that location's scene
);

CREATE TABLE decoration_catalog (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    unlock_condition TEXT NOT NULL,   -- e.g. "pantry_items_5"
    sprite_path TEXT NOT NULL
);

-- Add to pantry_items:
ALTER TABLE pantry_items ADD COLUMN slot_index INTEGER;
```

## Decisions Made

1. **Maps**: Generated as TypeScript constants (no Tiled editor). Iterate by editing code + hot reload.
2. **Tilesets**: Kenney Kitchen Tiles CC0 — downloaded by `scripts/download_kitchen_assets.py`.
3. **Decoration sprites**: Kenney CC0 packs (Platformer Pack Extended, Food Kit) — same download script.
4. **Food icon atlas**: Built by `scripts/build_phaser_atlas.py` from K1's Fluent Emoji PNGs — fully automated.
5. **Phaser version**: Phaser 3 (stable). Phaser 4 is still beta as of 2026-03.
6. **Slot persistence**: Single-user for now — no sync concerns.
